For example with SharpSnmpLib:
csharpvar result = await Messenger.GetAsync(...);

foreach (var binding in result)
{
    var oid   = binding.Id.ToString();        // "1.3.6.1.2.1.1.5.0"
    var type  = binding.Data.TypeCode;        // SnmpType.OctetString
    var value = binding.Data.ToString();      // "core-switch-01"
}
```

What you get back per type:
```
OctetString   → string   "core-switch-01"  or  "GigabitEthernet0/1"
IpAddress     → string   "192.168.1.1"
OID           → string   "1.3.6.1.2.1.1.1"
They are all just .ToString() — a plain string you can log, store in a DB, or display in a table. There is no numeric component at all.
This confirms the simpler approach: these values have no place in Prometheus. You would handle them in a separate MediatR handler that writes to a database or log, while the OtelMetricHandler only processes the numeric types.

In Prometheus, raw_value is just a label. Grafana reads labels as columns regardless of what is in them. So a table query like:
promqlsnmp_oid{agent="192.168.1.1"}
```

Produces columns for every label plus the numeric value column:
```
oid                       human_name        agent          source  raw_value          Value
─────────────────────────────────────────────────────────────────────────────────────────────
1.3.6.1.2.1.25.3.3.1.2   hrProcessorLoad   192.168.1.1    poll    73                 73
1.3.6.1.2.1.1.5.0         sysName           192.168.1.1    poll    core-switch-01     1
1.3.6.1.2.1.1.4.0         sysContact        192.168.1.1    poll    admin@company.com  1
The numeric Value column is meaningless for strings — you just hide it in Grafana. The actual information is in raw_value.
So the approach of Gauge = 1 with the string in raw_value is perfectly usable for a Grafana table. You get everything in one place — numbers and strings — with the same query and the same panel.

The raw_value label for numbers is just a convenience copy of the same value so the table panel can show it in a column without needing to reference the gauge value separately. It is redundant — the real data is the 73 at the end.
For strings it is the opposite — the gauge value 1 is meaningless, and raw_value is where the actual data lives.
snmp_oid{oid="...", human_name="sysName", raw_value="core-switch-01"} 1
                                          ────────────────────────    ─
                                          actual data                 meaningless
So raw_value serves different purposes depending on the type — redundant for numbers, essential for strings. If you want to keep the design clean you could drop raw_value for numbers entirely and only set it for non-numeric types. The gauge value is always there for numbers anyway.

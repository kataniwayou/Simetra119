  **41-sns-01-not-ready.sh -- Not Ready state**

  This script proves that a freshly loaded tenant is "not ready" during the readiness grace window.

  Structure:
  1. Save current tenants ConfigMap
  2. Apply tenant-cfg05-four-tenant-snapshot.yaml
  3. Wait for tenant vector reload log (poll_until_log 60 5)
  4. DO NOT prime (no sim_set_oid, no sleep 8). The test must assert BEFORE grace window expires.
  5. Immediately poll for the "not ready" log pattern within a short window:
     - Pattern: `"e2e-tenant-G1-T1.*not ready"` (LogDebug from SnapshotJob)
     - Use poll_until_log with timeout=15s, interval=1s, since=15s
     - With 1s SnapshotJob interval, the first cycle fires within ~1s of reload. It will log "not ready" because ReadinessGrace = 3*1*2.0 = 6s and no data has arrived.
  6. record_pass/record_fail for "SNS-01: G1-T1 not ready during grace window"
  7. Cleanup: reset_oid_overrides, restore_configmap

  Note: This script does NOT need to prime other tenants. The point is to catch the "not ready" log before grace expires. The 1s SnapshotJob interval ensures rapid cycling.

  **42-sns-02-stale-to-commands.sh -- Stale to Commands**

  This script proves that stale OIDs cause tier=1 skip-to-commands (tier=4).

  Structure:
  1. Save + apply tenant-cfg05-four-tenant-snapshot.yaml, wait for reload
  2. Prime T1 with valid data (required for readiness grace + so data ages out):
     ```bash
     sim_set_oid "4.1" "10"   # T1 eval in-range (>= Min:10)
     sim_set_oid "4.2" "1"    # T1 res1 in-range (>= Min:1)
     sim_set_oid "4.3" "1"    # T1 res2 in-range (>= Min:1)
     ```
  3. Sleep 8s (grace=6s + margin) so T1 becomes ready
  4. Capture sent counter baseline BEFORE stale switch (must precede stale OID changes so counter reflects only post-stale traffic):
     - `BEFORE_SENT=$(snapshot_counter "snmp_command_sent_total" 'device_name="E2E-SIM"')`
  5. Switch T1 OIDs to stale:
     ```bash
     sim_set_oid_stale "4.1"
     sim_set_oid_stale "4.2"
     sim_set_oid_stale "4.3"
     ```
  6. Wait for staleness age-out. With IntervalSeconds=1 and GraceMultiplier=2.0, staleness = 1 * 2.0 = 2s from newest sample. The data was fresh 0s ago when we set stale, so after ~3-4s the newest sample will be >2s old. Sleep 5s for margin.
  7. Poll for tier=1 stale log:
     - Pattern: `"e2e-tenant-G1-T1.*tier=1 stale"`
     - poll_until_log 30 1 ... since=15
  8. Also poll for tier=4 commands enqueued (stale skips to commands):
     - Pattern: `"e2e-tenant-G1-T1.*tier=4 -- commands enqueued"`
     - poll_until_log 30 1 ... since=15
  9. Verify sent counter increment (proves command-holder was evaluated despite stale):
     - `poll_until 30 2 "snmp_command_sent_total" 'device_name="E2E-SIM"' "$BEFORE_SENT"`
  10. record_pass/record_fail for each sub-assertion (42a: tier=1 stale log, 42b: tier=4 commands log, 42c: sent counter increment)
  11. Cleanup: reset_oid_overrides, restore_configmap

  IMPORTANT for both scripts:
  - Use `|| true` after save_configmap, kubectl apply, poll_until_log (within if block)
  - Follow exact pattern from mts-03-starvation-proof.sh for save/restore flow
  - All log patterns must include tenant name "e2e-tenant-G1-T1"
  - Use the `FIXTURES_DIR` pattern: `FIXTURES_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)/fixtures"`
  </action>

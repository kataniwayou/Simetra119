Create `tests/e2e/run-stage2.sh` -- the PSS Stage 2 runner that implements PSS-INF-01 (stage gating).

The runner must:
1. Be a standalone executable bash script (#!/usr/bin/env bash, set -euo pipefail)
2. Source all libraries (common.sh, prometheus.sh, kubectl.sh, report.sh, sim.sh) -- same as run-all.sh
3. Set up cleanup trap with stop_port_forwards (same pattern as run-all.sh)
4. Start port-forwards (prometheus 9090, e2e-simulator 8080) -- same as run-all.sh
5. Run pre-flight checks (check_pods_ready, check_prometheus_reachable) -- same as run-all.sh
6. Source Stage 1 scenarios (53-58) sequentially -- this runs them and accumulates FAIL_COUNT
7. **GATE CHECK:** After Stage 1 scenarios complete, check FAIL_COUNT:
   - If FAIL_COUNT > 0: print "Stage 1 had $FAIL_COUNT failure(s) -- skipping Stage 2 scenarios" and generate report, then exit 1
   - If FAIL_COUNT == 0: print "Stage 1 passed -- proceeding to Stage 2 scenarios"
8. Source Stage 2 scenarios (59-61) sequentially
9. Generate report and print summary
10. Exit with `[ "$FAIL_COUNT" -eq 0 ]` (same as run-all.sh)

The Stage 1 scenarios are sourced by globbing `scenarios/5[345678]-pss-*.sh` (scenarios 53-58).
The Stage 2 scenarios are sourced by globbing `scenarios/5[9]-pss-*.sh` and `scenarios/6[01]-pss-*.sh` (scenarios 59-61).

Alternatively, source them explicitly:
- Stage 1: source scenarios 53, 54, 55, 56, 57, 58
- Stage 2: source scenarios 59, 60, 61

Use explicit sourcing (list each file) for clarity and to avoid picking up non-PSS scenarios.

Banner should say "PSS Stage 2 Runner" to distinguish from run-all.sh.

The report file should be named `e2e-pss-stage2-report-YYYYMMDD-HHMMSS.md`.

IMPORTANT: The runner does NOT re-run all scenarios 1-52. It ONLY runs PSS scenarios (53-61). Non-PSS scenarios are run by run-all.sh separately.

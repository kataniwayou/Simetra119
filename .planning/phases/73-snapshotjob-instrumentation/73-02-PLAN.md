    **Part A: Add private counting methods** (after the existing private static methods at the bottom of the class)

    These mirror the existing tier check methods but return counts instead of booleans. They use the same exclusion/filtering logic as the existing check methods.

    1. `CountStaleHolders(IReadOnlyList<MetricSlotHolder> holders)` — returns int.
       Count holders that are stale using the same logic as `HasStaleness`:
       - Skip holders with Source=Trap, Source=Command, or IntervalSeconds=0
       - For remaining holders: ReadSlot() is null = stale, or age > graceWindow = stale
       - Unlike HasStaleness, do NOT short-circuit — count ALL stale holders
       - Return the total count

    2. `CountResolvedNonViolated(IReadOnlyList<MetricSlotHolder> holders)` — returns int.
       Count Resolved-role holders that are NOT violated (the inverse of AreAllResolvedViolated):
       - Only check holders with Role == "Resolved"
       - For Trap/Command sources: check newest sample only (ReadSlot). If null, skip. If not violated, count it.
       - For Poll/Synthetic: check all series samples. If ANY sample is not violated, count the holder as non-violated.
       - Holders with empty series (Length == 0) do not participate.
       - Return count of non-violated holders.

    3. `CountEvaluateViolated(IReadOnlyList<MetricSlotHolder> holders)` — returns int.
       Count Evaluate-role holders that ARE violated:
       - Only check holders with Role == "Evaluate"
       - For Trap/Command sources: check newest sample only. If null, skip. If violated, count it.
       - For Poll/Synthetic: ALL samples in the series must be violated for the holder to count as violated.
       - Holders with empty series do not participate.
       - Return count of violated holders.

    **Part B: Add RecordAndReturn helper**

    Private method that records state gauge + duration and returns the TenantState. This ensures every return point records both metrics and reduces repetition.

    ```csharp
    private TenantState RecordAndReturn(Tenant tenant, TenantState state, Stopwatch sw)
    {
        _tenantMetrics.RecordTenantState(tenant.Id, tenant.Priority, state);
        _tenantMetrics.RecordEvaluationDuration(tenant.Id, tenant.Priority, sw.Elapsed.TotalMilliseconds);
        return state;
    }
    ```

    **Part C: Instrument EvaluateTenant with Stopwatch and metric calls**

    Modify `EvaluateTenant` as follows:

    1. Add `var sw = Stopwatch.StartNew();` as the FIRST line of the method.

    2. **NotReady path** (pre-tier, currently line 140):
       - NO tier counters, NO command counters (per CONTEXT.md decision)
       - Record state gauge + duration only
       - Replace `return TenantState.NotReady;` with `return RecordAndReturn(tenant, TenantState.NotReady, sw);`

    3. **Resolved path** (tier 2, currently line 159):
       - Record tier counters: count stale holders and call IncrementTier1Stale once per stale holder in a loop. Count resolved non-violated and call IncrementTier2Resolved once per non-violated holder in a loop.
       - Pattern for tier counter recording (add before the return):
         ```csharp
         var staleCount = CountStaleHolders(tenant.Holders);
         for (var i = 0; i < staleCount; i++)
             _tenantMetrics.IncrementTier1Stale(tenant.Id, tenant.Priority);
         var resolvedCount = CountResolvedNonViolated(tenant.Holders);
         for (var i = 0; i < resolvedCount; i++)
             _tenantMetrics.IncrementTier2Resolved(tenant.Id, tenant.Priority);
         ```
       - Replace `return TenantState.Resolved;` with `return RecordAndReturn(tenant, TenantState.Resolved, sw);`

    4. **Healthy path** (tier 3, currently line 173):
       - Record ALL three tier counters: stale + resolved + evaluate (even when counts are 0)
       - Pattern (add before the return):
         ```csharp
         var staleCount = CountStaleHolders(tenant.Holders);
         for (var i = 0; i < staleCount; i++)
             _tenantMetrics.IncrementTier1Stale(tenant.Id, tenant.Priority);
         var resolvedCount = CountResolvedNonViolated(tenant.Holders);
         for (var i = 0; i < resolvedCount; i++)
             _tenantMetrics.IncrementTier2Resolved(tenant.Id, tenant.Priority);
         var evaluateCount = CountEvaluateViolated(tenant.Holders);
         for (var i = 0; i < evaluateCount; i++)
             _tenantMetrics.IncrementTier3Evaluate(tenant.Id, tenant.Priority);
         ```
       - Replace `return TenantState.Healthy;` with `return RecordAndReturn(tenant, TenantState.Healthy, sw);`

    5. **Unresolved path** (tier 4, currently line 216):
       - Record ALL three tier counters (same pattern as Healthy, add before the command dispatch loop)
       - In the command dispatch loop, ADD `_tenantMetrics.IncrementCommand*` calls alongside existing `_pipelineMetrics.IncrementCommand*` calls:
         - Suppressed: `_tenantMetrics.IncrementCommandSuppressed(tenant.Id, tenant.Priority);` alongside existing `_pipelineMetrics.IncrementCommandSuppressed(tenant.Id);`
         - Dispatched: `_tenantMetrics.IncrementCommandDispatched(tenant.Id, tenant.Priority);` alongside existing `_pipelineMetrics.IncrementCommandDispatched(tenant.Id);`
         - Failed (channel full): `_tenantMetrics.IncrementCommandFailed(tenant.Id, tenant.Priority);` alongside existing `_pipelineMetrics.IncrementCommandFailed(tenant.Id);`
       - Replace `return TenantState.Unresolved;` with `return RecordAndReturn(tenant, TenantState.Unresolved, sw);`

    **Important design notes:**
    - The tier counter recording block (stale/resolved/evaluate counting + loops) should be computed ONCE and placed before the return at each applicable exit point. For Healthy and Unresolved paths that share the same counters, the counting happens at different points in the method — do NOT factor it into a shared call (each return path is explicit).
    - Keep existing `_pipelineMetrics.IncrementCommand*` calls — they are additive, not replacements.
    - The stale count block goes AFTER the HasStaleness check (the if/else determines flow, counting is for metrics only).
    - For the stale->tier4 path (HasStaleness=true falls through to tier 4), the tier counters go before the command dispatch loop.

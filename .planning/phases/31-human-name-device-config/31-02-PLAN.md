     - **Mutate poll.Names in place:** `poll.Names.Clear(); poll.Names.AddRange(resolvedOids);`
       - Comment: "// Replace metric names with resolved OID strings -- DeviceRegistry reads these as OIDs"
       - This works because the deserialized List<DeviceOptions> is a fresh object per reload
   - Return the mutated devices list (same reference)

**DeviceRegistry changes:**

4. Add `IOidMapService _oidMapService` field. Inject via constructor (add parameter `IOidMapService oidMapService` after `ILogger<DeviceRegistry> logger`). Assign to field.

5. In the constructor, BEFORE building poll groups from `d.Polls`, add name resolution:
   ```csharp
   // Resolve metric names to OIDs at startup (same logic as DeviceWatcherService reload path)
   foreach (var d in devices)
   {
       foreach (var poll in d.Polls)
       {
           var resolvedOids = new List<string>(poll.Names.Count);
           foreach (var name in poll.Names)
           {
               var oid = _oidMapService.ResolveToOid(name);
               if (oid is not null)
               {
                   resolvedOids.Add(oid);
               }
               else
               {
                   _logger.LogWarning(
                       "Metric name '{MetricName}' not found in OID map for device {DeviceName}",
                       name, d.Name);
               }
           }
           poll.Names.Clear();
           poll.Names.AddRange(resolvedOids);
       }
   }
   ```

6. In both constructor and ReloadAsync, the existing code reads `poll.Names.AsReadOnly()` to build PollInfo.Oids. After resolution, poll.Names contains resolved OID strings, so this naturally flows into PollInfo(Oids: poll.Names.AsReadOnly()). No change needed to the PollInfo construction line itself.

7. **Empty poll group handling:** After resolution, a poll group might have zero resolved names. The existing code builds PollInfo with an empty Oids list. DynamicPollScheduler.ReconcileAsync should handle this naturally (a poll job with zero OIDs either does nothing or isn't scheduled). Verify by reading DynamicPollScheduler -- if it creates a job regardless, add a filter: skip poll groups where `poll.Names.Count == 0` when building pollGroups in both constructor and ReloadAsync:
   ```csharp
   var pollGroups = d.Polls
       .Select((poll, index) => new PollInfo(
           PollIndex: index,
           Oids: poll.Names.AsReadOnly(),
           IntervalSeconds: poll.IntervalSeconds,
           TimeoutMultiplier: poll.TimeoutMultiplier))
       .Where(pi => pi.Oids.Count > 0)  // Skip groups with zero resolved names
       .ToList()
       .AsReadOnly();
   ```
   Add this `.Where` filter in BOTH the constructor and ReloadAsync.

8. **Device always registered:** Even if ALL poll groups have zero resolved names (all filtered out), the device is still added to `byIpPortBuilder` and `byNameBuilder`. The device needs to exist for trap routing. The existing code already does this (DeviceInfo is created regardless of poll groups).

   ```csharp

   /// Reverse-resolves a metric name to its OID.
   /// Returns null if the metric name is not in the current map.
   /// </summary>
   string? ResolveToOid(string metricName);
   ```

2. In `OidMapService.cs`, add a private volatile field:
   ```csharp
   private volatile FrozenDictionary<string, string> _reverseMap = FrozenDictionary<string, string>.Empty;
   ```

3. Add a static `BuildReverseMap` method:
   ```csharp
   private static FrozenDictionary<string, string> BuildReverseMap(FrozenDictionary<string, string> forwardMap)
   {
       return forwardMap
           .Select(kv => new KeyValuePair<string, string>(kv.Value, kv.Key))
           .ToFrozenDictionary(StringComparer.Ordinal);
   }
   ```
   Use `StringComparer.Ordinal` for reverse map keys (metric names are case-sensitive lowercase snake_case in this codebase).

4. In the constructor, after `_metricNames = _map.Values.ToFrozenSet();`, add:
   ```csharp
   _reverseMap = BuildReverseMap(_map);
   ```
   IMPORTANT: Build from `_map` (after `MergeWithHeartbeatSeed`), NOT from `initialEntries`. This ensures the heartbeat seed is in the reverse index.

5. In `UpdateMap`, after `_metricNames = newMap.Values.ToFrozenSet();`, add:
   ```csharp
   _reverseMap = BuildReverseMap(newMap);
   ```
   Same principle: build from `newMap` (post-heartbeat-seed), not from `entries`.

6. Implement `ResolveToOid`:
   ```csharp
   public string? ResolveToOid(string metricName)
   {
       return _reverseMap.TryGetValue(metricName, out var oid) ? oid : null;
   }
   ```

Note: The two volatile writes (`_map`/`_metricNames` then `_reverseMap`) are NOT truly atomic, but this matches the existing pattern where `_map` and `_metricNames` already have the same brief inconsistency window. Do NOT introduce locking — it would diverge from the established codebase pattern.
  </action>

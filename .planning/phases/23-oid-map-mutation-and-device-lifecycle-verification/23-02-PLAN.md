**1. `tests/e2e/fixtures/device-added-configmap.yaml` (DEV-01)**

Full copy of fake-device-configmap.yaml with ONE addition -- a new device entry appended after FAKE-UNREACHABLE:
```json
{
    "Name": "E2E-SIM-2",
    "IpAddress": "e2e-simulator.simetra.svc.cluster.local",
    "Port": 161,
    "CommunityString": "Simetra.E2E-SIM",
    "MetricPolls": [
        {
            "IntervalSeconds": 10,
            "Oids": [
                "1.3.6.1.4.1.47477.999.1.1.0"
            ]
        }
    ]
}
```

Key design decisions:
- Name "E2E-SIM-2" so it's a distinct device in Prometheus labels
- Same IP as E2E-SIM (same simulator pod) since this is just a logical device identity
- **Explicit `CommunityString: "Simetra.E2E-SIM"`** -- the E2E simulator only accepts this community string, but default convention would derive "Simetra.E2E-SIM-2" from the device name, which would be rejected. The CommunityString field overrides the convention (see MetricPollJob.cs lines 83-85).
- Polls only 1 OID (.999.1.1.0) -- minimal footprint, enough to prove device addition works

**2. `tests/e2e/scenarios/22-device-remove.sh` (DEV-02: Remove Device)**

```
# Scenario 22: Removing a device stops poll metrics
# Removes E2E-SIM from devices ConfigMap, waits for poll scheduler to stop,
# then verifies poll_executed counter delta = 0 over a 20s window.
```

Flow:
1. `snapshot_configmaps`
2. Snapshot `snmp_poll_executed_total` with filter `device_name="E2E-SIM"` -- establish pre-removal baseline
3. `kubectl apply -f "$SCRIPT_DIR/fixtures/device-removed-configmap.yaml" -n simetra`
4. Wait 20s (allow watcher detection ~5s + poll interval 10s + OTel export 15s to flush any in-flight data)
5. Snapshot `snmp_poll_executed_total` again for E2E-SIM -- this is the "post-removal start" value
6. Wait 20s (2 poll intervals -- if device were still being polled, counter would increase)
7. Snapshot `snmp_poll_executed_total` again for E2E-SIM -- this is the "post-removal end" value
8. Compute DELTA = end - start (should be 0 since device is no longer polled)
9. If DELTA equals 0: `record_pass` with evidence
10. If DELTA > 0: `record_fail` -- device is still being polled after removal
11. `restore_configmaps`
12. Poll with 30s deadline for E2E-SIM poll_executed to start incrementing again (confirms restoration)
13. Log restoration confirmation

NOTE: No `assert_delta_gt` here -- we need the OPPOSITE assertion (delta = 0). Use inline comparison:
```bash
if [ "$DELTA" -eq 0 ]; then
    record_pass "$SCENARIO_NAME" "..."
else
    record_fail "$SCENARIO_NAME" "..."
fi
```

**3. `tests/e2e/scenarios/23-device-modify-interval.sh` (DEV-03: Modify Poll Interval)**

```
# Scenario 23: Modifying poll interval changes collection frequency
# Measures poll_executed delta at 10s interval (baseline), then halves to 5s,
# measures again. The faster interval should produce a higher delta.
```

Flow:
1. `snapshot_configmaps`
2. **Baseline measurement (10s interval):**
   - Snapshot `snmp_poll_executed_total` for `device_name="E2E-SIM"`
   - Wait 30s
   - Snapshot again, compute DELTA_BEFORE = after - before
   - Log: "Baseline delta at 10s interval: $DELTA_BEFORE"
3. `kubectl apply -f "$SCRIPT_DIR/fixtures/device-modified-interval-configmap.yaml" -n simetra`
4. Wait 20s for watcher + scheduler reconciliation + OTel buffer flush
5. **Fast measurement (5s interval):**
   - Snapshot `snmp_poll_executed_total` for `device_name="E2E-SIM"`
   - Wait 30s
   - Snapshot again, compute DELTA_AFTER = after - before
   - Log: "Fast delta at 5s interval: $DELTA_AFTER"
6. Verify DELTA_AFTER > DELTA_BEFORE (faster interval = more polls in same window)
7. `assert_delta_gt "$DELTA_AFTER" "$DELTA_BEFORE" "$SCENARIO_NAME" "delta_10s=$DELTA_BEFORE delta_5s=$DELTA_AFTER"`
8. `restore_configmaps`
9. Log restoration confirmation

**Pattern notes for all scenarios:**
- Scenario 22 is the longest (total ~60s of waiting). This is necessary for reliable stagnation detection.
- Scenario 23 is also long (~80s of measurement windows). This is necessary for statistical reliability.
- Follow the exact patterns from scenarios 01 and 15 for counter assertions and ConfigMap operations.

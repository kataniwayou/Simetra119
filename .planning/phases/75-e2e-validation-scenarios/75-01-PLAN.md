Create `tests/e2e/scenarios/108-tvm02-notready.sh` — verifies the NotReady evaluation path.

**Key design (from RESEARCH.md pitfall 1 and 3):**
- NotReady is triggered by applying the fixture and querying IMMEDIATELY — do NOT prime OIDs, do NOT sleep.
- The tenant is in the readiness grace window; SnapshotJob returns TenantState.NotReady (0).
- NotReady path records ONLY state gauge + duration. No tier counters, no command counters.

**Scenario structure:**
1. Apply `tenant-cfg06-pss-single.yaml` fixture (save original ConfigMap first).
2. Wait for tenant vector reload via `poll_until_log`.
3. Do NOT prime OIDs. Do NOT sleep 8s. The tenant has no poll data yet — it is NotReady.
4. Wait briefly (sleep 3-5s) for at least one SnapshotJob cycle to evaluate the tenant and record metrics.

**Sub-assertions:**

TVM-02A: `tenant_state` gauge equals 0 (NotReady)
```bash
# Poll for tenant_state to appear (newly applied tenant may need one cycle)
poll_until_exists 30 3 'tenant_state{tenant_id="e2e-pss-tenant"}'
STATE=$(query_prometheus 'tenant_state{tenant_id="e2e-pss-tenant"}' | jq -r '.data.result[0].value[1] // "-1"' | cut -d. -f1)
if [ "$STATE" = "0" ]; then
    record_pass "TVM-02A: tenant_state=0 (NotReady) during grace window" "state=${STATE}"
else
    record_fail "TVM-02A: tenant_state=0 (NotReady) during grace window" "state=${STATE} expected=0"
fi
```

TVM-02B: `tenant_evaluation_duration_milliseconds_count` > 0 (duration is recorded even for NotReady)
```bash
DURATION_COUNT=$(query_prometheus 'tenant_evaluation_duration_milliseconds_count{tenant_id="e2e-pss-tenant"}' | jq -r '.data.result[0].value[1] // "0"' | cut -d. -f1)
if [ "$DURATION_COUNT" -gt 0 ]; then
    record_pass "TVM-02B: duration histogram recorded during NotReady" "count=${DURATION_COUNT}"
else
    record_fail "TVM-02B: duration histogram recorded during NotReady" "count=${DURATION_COUNT} expected>0"
fi
```

TVM-02C: Tier counters do NOT increment (negative assertion). Snapshot `tenant_tier1_stale_total`, `tenant_tier2_resolved_total`, `tenant_tier3_evaluate_total` with filter `tenant_id="e2e-pss-tenant"` BEFORE the fixture is applied (or just after reload, before evaluation). Take baseline, sleep 10s, take after, assert all deltas == 0. Use `snapshot_counter` and `assert_delta_eq` for each.

**IMPORTANT:** The NotReady scenario is timing-sensitive. The tenant must be queried within the grace window. If using `poll_until_exists` introduces delay, the tenant may have exited the grace window. Consider that `poll_until_exists` with timeout=30 and interval=3 gives enough retries to catch the first cycle while the tenant is still new.

**Alternative approach if timing is too tight:** Instead of relying on grace window timing for TVM-02A, apply the fixture, poll until `tenant_state` series appears, and check the initial value. If it has already moved to Healthy (1), the grace window was too fast. In that case, use a fresh fixture name or re-apply after removing the tenant. However, the research says "apply and query immediately" works, so try that first.

**Cleanup:** `reset_oid_overrides`, then `restore_configmap`.

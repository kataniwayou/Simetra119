```
# Scenario 32: STS-04 Suppression window -- sent, then suppressed, then sent again after expiry
# Uses tenant-cfg01-suppression.yaml with SuppressionWindowSeconds=30
```

**CRITICAL DESIGN DECISION:** This script uses `tenant-cfg01-suppression.yaml` (NOT tenant-cfg01-single.yaml) because that fixture has `SuppressionWindowSeconds: 30` which is greater than the SnapshotJob interval of 15s. With the default 10s window, the suppression would expire before the next cycle fires, making it impossible to observe.

The suppressed counter uses label `device_name="e2e-tenant-A-supp"` (the tenant ID from the suppression fixture).

**Steps:**

**Setup:**
1. save_configmap for simetra-tenants
2. `kubectl apply -f "$FIXTURES_DIR/tenant-cfg01-suppression.yaml"` (applies the 30s suppression window fixture)
3. poll_until_log 60 5 for "Tenant vector reload complete" (since 30s)
4. `sim_set_scenario command_trigger`

**Window 1 -- First command (SENT):**
5. Baseline: snapshot `snmp_command_sent_total{device_name="E2E-SIM"}` as `BEFORE_SENT_W1`
6. Wait for tier=4 log: `poll_until_log 90 5 "tier=4 — commands enqueued" 60`
7. Sub-scenario 32a: Assert tier=4 log found (record_pass/record_fail)
8. Snapshot `snmp_command_sent_total{device_name="E2E-SIM"}` as `AFTER_SENT_W1`
9. Sub-scenario 32b: Assert `AFTER_SENT_W1 - BEFORE_SENT_W1 > 0` using assert_delta_gt

**Window 2 -- Suppressed (WITHIN 30s window):**
10. Immediately baseline suppressed counter: snapshot `snmp_command_suppressed_total{device_name="e2e-tenant-A-supp"}` as `BEFORE_SUPP_W2`
11. Also baseline sent counter: snapshot `snmp_command_sent_total{device_name="E2E-SIM"}` as `BEFORE_SENT_W2`
12. Wait for next SnapshotJob cycle to fire another evaluation. The suppression window (30s) has NOT expired yet (only ~15s since first command). Use `poll_until 30 5 "snmp_command_suppressed_total" 'device_name="e2e-tenant-A-supp"' "$BEFORE_SUPP_W2"` to detect the suppressed counter incrementing.
    - If poll_until succeeds: the suppression cache blocked the command.
    - Note: poll_until from prometheus.sh polls until metric > baseline.
13. Sub-scenario 32c: Assert suppressed counter incremented (`snapshot - BEFORE_SUPP_W2 > 0`) using assert_delta_gt.
14. Sub-scenario 32d: Assert sent counter did NOT increment during Window 2 (`snapshot - BEFORE_SENT_W2 == 0`). Use record_pass/record_fail with direct arithmetic.

**Window 3 -- Sent again (window EXPIRED):**
15. Wait for suppression window to expire. The window is 30s from the first command. By this point, ~30-45s have elapsed from setup. Sleep for the remaining time to ensure the window expires. Use `sleep 20` as a safety margin to guarantee the 30s window is fully expired.
16. Baseline: snapshot `snmp_command_sent_total{device_name="E2E-SIM"}` as `BEFORE_SENT_W3`
17. Wait for next tier=4: `poll_until_log 60 5 "tier=4 — commands enqueued" 30` (since 30s to only see recent logs)
18. Sub-scenario 32e: Assert tier=4 log found again
19. Snapshot sent counter as `AFTER_SENT_W3`
20. Sub-scenario 32f: Assert `AFTER_SENT_W3 - BEFORE_SENT_W3 > 0` using assert_delta_gt

**Cleanup:**
21. reset_scenario
22. restore_configmap from saved snapshot

**Key warnings:**
- Do NOT use tenant-cfg01-single.yaml for this test (10s window < 15s interval = impossible to observe suppression)
- The suppressed counter label value is the TENANT ID from the fixture (`e2e-tenant-A-supp`), not the device name
- Window 2 assertion must happen BEFORE the 30s suppression window expires
- The sleep in Window 3 is the ONLY fixed sleep in the entire Phase 53 suite -- it's unavoidable because we must wait for the window to expire, and there is no log event for window expiry

```
# Scenario 33: STS-05 Staleness detection -- tier=1 stale, no command counters
```

**Steps:**

**Setup:**
1. save_configmap for simetra-tenants
2. `kubectl apply -f "$FIXTURES_DIR/tenant-cfg01-single.yaml"` (standard single-tenant fixture)
3. poll_until_log 60 5 for "Tenant vector reload complete" (since 30s)
4. First, set a NON-stale scenario to establish fresh poll data: `sim_set_scenario healthy`
5. Wait for at least one successful poll cycle to populate fresh timestamps: `sleep 20` (this ensures the holders have recent data so staleness detection has something to age out)

**Switch to stale:**
6. Baseline: snapshot `snmp_command_sent_total{device_name="E2E-SIM"}` and `snmp_command_suppressed_total{device_name="e2e-tenant-A"}` BEFORE
7. `sim_set_scenario stale` (returns NoSuchInstance for .4.1 and .4.2 -- polls fail, timestamps stop updating)

**Wait for staleness:**
8. The grace window is `IntervalSeconds(10) * GraceMultiplier(2.0) = 20s`. After 20s of stale data + one SnapshotJob cycle (15s), the tier=1 log should appear.
9. `poll_until_log 90 5 "tier=1 stale" 60` -- generous timeout to accommodate timing variance.

**Assertions:**
10. Sub-scenario 33a (log assertion): record_pass/record_fail based on poll_until_log result for "STS-05: Staleness tier=1 log detected"
11. Sub-scenario 33b (counter assertion): Snapshot AFTER. Assert `snmp_command_sent_total` delta == 0 (no commands while stale). Use record_pass/record_fail with direct arithmetic.
12. Sub-scenario 33c (counter assertion): Assert `snmp_command_suppressed_total` delta == 0 (no suppression events while stale either).

**Cleanup:**
13. reset_scenario (switches back to default, polls start succeeding again)
14. restore_configmap from saved snapshot

**Key details:**
- The stale scenario causes NoSuchInstance for .4.1 and .4.2. The existing poll timestamps age out past the grace window.
- The grep pattern for poll_until_log is `"tier=1 stale"` which matches the SnapshotJob.cs log: `"tier=1 stale — skipping threshold checks"`
- Use tenant-cfg01-single.yaml (NOT the suppression fixture) -- standard fixture is correct here
- The `sim_set_scenario healthy` + sleep 20 priming step ensures the holder slots have recent timestamps that can then age out. Without this, slots might be null (never polled) and `HasStaleness` returns false for null slots.

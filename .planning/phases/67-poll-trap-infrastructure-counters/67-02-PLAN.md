**80-mcv11-poll-unreachable.sh** (copy from scenario 06 with MCV label):
- SCENARIO_NAME: "MCV-11: poll.unreachable increments after 3 consecutive failures"
- METRIC: `snmp_poll_unreachable_total`
- FILTER: `device_name="FAKE-UNREACHABLE"`
- Copy the ENTIRE implementation from scenario 06 verbatim, including:
  1. Save original devices ConfigMap: `save_configmap "simetra-devices" "simetra" "$ORIGINAL_CM"`
  2. Pre-recovery step (MANDATORY for idempotency): Build ConfigMap with FAKE-UNREACHABLE at reachable IP (e2e-simulator.simetra.svc.cluster.local), apply it, sleep 20s for tracker reset
  3. Snapshot BEFORE
  4. Apply `fake-device-configmap.yaml` (FAKE-UNREACHABLE at 10.255.255.254, IntervalSeconds=10)
  5. poll_until 120s for unreachable counter to increment (3 failures * 10s interval + 8s timeout per failure + OTel lag)
  6. Snapshot AFTER, compute DELTA
  7. assert_delta_gt DELTA 0
- CRITICAL: The pre-recovery step MUST be preserved. The DeviceUnreachabilityTracker is a singleton that persists across device config changes. Without the pre-recovery step, if FAKE-UNREACHABLE was left in unreachable state from a previous E2E run, the transition counter won't fire again.
- Use the same ORIGINAL_CM variable name and path pattern as scenario 06: `ORIGINAL_CM="$SCRIPT_DIR/fixtures/.original-devices-configmap.yaml"`
- The SCRIPT_DIR variable may need to be set. Check how scenario 06 accesses it -- if it uses `$(dirname "${BASH_SOURCE[0]}")`, use the same pattern. Actually, scenarios are sourced by run-all.sh which sets SCRIPT_DIR. Verify by reading scenario 06's usage.
- The `Polls` field in the pre-recovery ConfigMap must use `Metrics` (not `MetricNames`): `[{"MetricName": "e2e_gauge_test"}]` wrapped in `"Metrics": [...]`. Check scenario 06's jq command for the exact format.
- Leave FAKE-UNREACHABLE in unreachable state at the end (do NOT restore ConfigMap here -- scenario 81 needs it unreachable for recovery test).

**81-mcv12-poll-recovered.sh** (copy from scenario 07 with MCV label):

**82-mcv13-tenantvector-routed.sh**:
- SCENARIO_NAME: "MCV-13: tenantvector.routed increments on fan-out write"
- METRIC: `snmp_tenantvector_routed_total`
- FILTER: `""` (empty -- sum across all device_name tags)

Pattern (belt-and-suspenders approach from research):
1. Unconditionally apply simetra-tenants.yaml to ensure tenants are active. This handles the case where scenario 28 restored an empty original ConfigMap. Use the same path resolution as scenario 28:
   ```bash
   TENANTVECTOR_YAML="$(cd "$(dirname "${BASH_SOURCE[0]}")/../../.." && pwd)/deploy/k8s/snmp-collector/simetra-tenants.yaml"
   ```
   The repo root is 3 levels up from tests/e2e/scenarios/ (scenarios -> e2e -> tests -> root). So `../../..` is correct.
2. kubectl apply -f "$TENANTVECTOR_YAML" (with `|| true` for safety)
3. Sleep 30s for the tenant watcher to reload and first poll cycle to complete routing
4. Snapshot BEFORE
5. poll_until 90s with 5s interval for the counter to increment
6. Snapshot AFTER, compute DELTA
7. assert_delta_gt DELTA 0 with evidence

Do NOT do a deployment restart (unlike scenario 28 which does a full rollout restart). The watcher detects ConfigMap changes via file system watch and hot-reloads. A restart is unnecessary and would add 90s+ of delay.

Do NOT restore the ConfigMap afterward. Scenario 28 already handles tenants ConfigMap lifecycle, and leaving tenants active is harmless for subsequent scenarios.

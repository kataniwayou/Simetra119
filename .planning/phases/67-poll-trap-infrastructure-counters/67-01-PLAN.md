- poll_until call: `poll_until 45 "$POLL_INTERVAL" "$METRIC" "$FILTER" "$BEFORE"`
- EVIDENCE string: include before, after, delta, and a note "(expect > 0: fires in finally block per poll group per replica)"

**77-mcv09-trap-received.sh** (model on scenario 04):
- SCENARIO_NAME: "MCV-09: trap.received increments for valid-community traps"
- METRIC: `snmp_trap_received_total`
- FILTER: `device_name="E2E-SIM"` (valid traps use community="Simetra.E2E-SIM")
- Pattern: snapshot_counter BEFORE, poll_until with explicit timeout, snapshot AFTER, assert_delta_gt DELTA 0
- poll_until call: `poll_until 60 "$POLL_INTERVAL" "$METRIC" "$FILTER" "$BEFORE"`
- ChannelConsumerService increments per varbind dequeued. E2E-SIM sends 1 varbind per trap. With 3 replicas receiving broadcast: delta >= 3 per trap cycle.
- EVIDENCE: before, after, delta, "(valid traps every 30s, 3 replicas)"

**78-mcv09b-trap-received-negative.sh** (new pattern, folded with auth_failed):
- SCENARIO_NAME: "MCV-09b: trap.received does not increment for bad-community traps"
- This proves that bad-community traps (dropped by SnmpTrapListenerService before reaching ChannelConsumerService) do not contribute to trap.received.
- Pattern:
  1. Snapshot both `snmp_trap_received_total{device_name="E2E-SIM"}` and `snmp_trap_auth_failed_total` (no filter, uses device_name="unknown")
  2. poll_until for auth_failed to increment (confirming a bad trap arrived; bad trap interval = 45s): `poll_until 75 "$POLL_INTERVAL" "snmp_trap_auth_failed_total" "" "$AUTH_FAILED_BEFORE"`
  3. sleep 15 for OTel export flush
  4. Snapshot both counters again
  5. If auth_failed delta > 0 (bad trap confirmed arrived): check that any trap.received increase during this window is attributable to valid traps (valid traps use device_name="E2E-SIM", bad traps never reach ChannelConsumerService at all). Since bad traps are dropped before channel write, they literally cannot increment trap.received for ANY device_name. Record pass with evidence showing auth_failed fired but trap.received only shows valid-community increments.
  6. If auth_failed delta == 0: record_fail (no bad trap arrived in window)
- IMPORTANT: The negative assertion is proven by mechanism (bad traps dropped before ChannelConsumerService), not by observing zero delta. Valid traps every 30s WILL cause trap.received to increment during the window. The proof is: auth_failed incremented (bad trap arrived) AND trap.received only has device_name="E2E-SIM" entries (no "unknown" or "BadCommunity" device name in trap.received).
- Use `query_counter "snmp_trap_received_total" 'device_name="unknown"'` to verify there is NO trap.received with device_name="unknown". If this returns 0 (via vector(0) fallback), it proves bad-community traps never reach ChannelConsumerService.

**2. Add the `AreAllReady` static method** (place near HasStaleness, before or after it):

```csharp
/// <summary>
/// Pre-tier: Checks whether all holders in the tenant are past their readiness grace window.
/// A holder with real data (from WriteValue or CopyFrom) is always considered ready,
/// even if its ConstructedAt is recent (handles config reload).
/// </summary>
private static bool AreAllReady(IReadOnlyList<MetricSlotHolder> holders)
{
    foreach (var holder in holders)
    {
        if (!holder.IsReady)
            return false;
    }
    return true;
}
```

Note: This delegates to `MetricSlotHolder.IsReady` which already handles the data-present short-circuit (from Plan 01).

**3. Update HasStaleness null-slot handling.**

In HasStaleness, change the null-slot behavior. Currently (line 224-225):
```csharp
if (slot is null)
    continue; // No data yet — cannot judge, not stale
```

Change to:
```csharp
if (slot is null)
    return true; // Grace ended, no data — device never responded — stale
```

This is safe because: `HasStaleness` is only reached AFTER `AreAllReady` returns true. If a holder has no data AND readiness passed, the device genuinely never responded. The `continue` behavior was originally for sentinels (which are now gone) — a null slot post-readiness means stale.

Update the XML doc comment on HasStaleness: change "Holders with null ReadSlot are skipped (cannot judge, not stale)" to "Holders with null ReadSlot after readiness check are stale (device never responded)."

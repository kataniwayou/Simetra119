**TierResult enum — exactly 4 states (verify or update from 48-02/03):**

```csharp
enum TierResult
{
    Stale,         // Tier 1 detected staleness — no threshold check, no commands
    ConfirmedBad,  // Tier 2: ALL Resolved violated — device already handled, evaluation stopped, no commands
    Healthy,       // Tier 3: NOT all Evaluate violated — device is fine, no commands
    Commanded      // Tier 4: Resolved NOT all violated + ALL Evaluate violated + at least one command enqueued
}
```

**How each state is reached through the correct sequential tier logic:**
- **Stale:** Tier 1 detects staleness. Evaluation stops. Return `TierResult.Stale`.
- **ConfirmedBad:** Tier 2 checks Resolved holders — ALL are violated. Device is in confirmed bad state (already handled/resolved). Evaluation STOPS here. No commands issued. Return `TierResult.ConfirmedBad`.
- **Healthy:** Tier 2 Resolved NOT all violated → continue to Tier 3. Tier 3 checks Evaluate holders — NOT all violated. Device is fine. Return `TierResult.Healthy`.
- **Commanded:** Tier 2 Resolved NOT all violated → continue to Tier 3. Tier 3 ALL Evaluate violated → Tier 4 enqueues commands. At least one `TryWrite` returned true. Return `TierResult.Commanded`.

**The commanding path IS reachable:**
Resolved NOT all violated (continue) → Evaluate ALL violated (command) → Tier 4 fires → TierResult.Commanded.

**Edge case — Tier 4 reached but zero TryWrite successes (all suppressed or channel full):**
Return `TierResult.ConfirmedBad` — device needs commanding but couldn't command this cycle. This is an advance-allowing state (same as ConfirmedBad from Tier 2 — no active commands firing, safe to cascade).

**Parallel evaluation within each group:**
- Use `Task.WhenAll` with `Select` over `group.Tenants` per RESEARCH.md Pattern 3
- Pre-allocate a result array with one slot per tenant (indexed by position) per RESEARCH.md recommendation — each task writes to its own index, no locking needed
- Do NOT use `foreach` with sequential `await` — this defeats parallelism

```csharp
var results = new TierResult[group.Tenants.Count];
await Task.WhenAll(group.Tenants.Select((tenant, index) =>
    EvaluateTenantAsync(tenant, index, results, ct)));
```

**Advance gate after each group:**
- After `await Task.WhenAll(...)`, inspect the results array
- **Advances when ALL tenants in group are either:**
  - `Healthy` — device is fine, no action needed, move on
  - `ConfirmedBad` — all Resolved violated (Tier 2 stop) OR all suppressed at Tier 4 — no active commands, move on
- **Blocks when ANY tenant is:**
  - `Stale` — data missing, uncertain state, don't cascade
  - `Commanded` — commands were actively issued this cycle, don't cascade to lower priority
- If advance gate blocks: `break` out of the groups loop — do NOT evaluate lower-priority groups

**Cycle summary logging (SNAP-15):**
After the groups loop (whether completed or broken), log at Debug level:
`"Snapshot cycle complete: {TenantsEvaluated} evaluated, {Commanded} commanded, {Stale} stale"`

Track these counters across all groups. Aggregate from the results arrays after each group's Task.WhenAll completes (simple counting, no locking needed).

**Liveness stamp and correlation ID** remain in the `finally` block (already placed by 48-01). Verify they still execute regardless of advance gate break.

## Issues Found

### Issue 1 (ACTIONABLE): PSS-17c and PSS-20c --since window not aligned with sleep

**Severity:** Flaky test risk -- same class as PSS-18c/19c fixed in Phase 65

**Files:**
- `tests/e2e/scenarios/65-pss-17-all-g1-unresolved.sh` line 104
- `tests/e2e/scenarios/68-pss-20-all-g1-not-ready.sh` line 108

**Description:** Phase 65 aligned PSS-18c and PSS-19c from --since=12s to --since=10s to match
the 10s observation sleep. PSS-17c and PSS-20c still use --since=12s with sleep 10. The 2-second
overlap allows prior-cycle G2 logs to appear inside the log absence window, causing intermittent
false-positive failures.

| Scenario | sleep | --since in code | Header comment | Status |
|----------|-------|----------------|----------------|--------|
| PSS-17c (65-pss-17) | 10s | 12s | 15s | MISMATCH -- 2s overlap risk, comment also wrong |
| PSS-18c (66-pss-18) | 10s | 10s | 10s | FIXED (Phase 65) |
| PSS-19c (67-pss-19) | 10s | 10s | 10s | FIXED (Phase 65) |
| PSS-20c (68-pss-20) | 10s | 12s | 12s | MISMATCH -- 2s overlap risk |

**Fix:** Change --since=12s to --since=10s in PSS-17c (line 104) and PSS-20c (line 108).
Also correct PSS-17c header comment from --since=15s to --since=10s.

### Issue 2 (DOCUMENTATION): PSS-20 64-VERIFICATION.md reflects original not-ready design

**File:** `.planning/phases/64-advance-gate-logic/64-VERIFICATION.md` truth row 11

**Description:** 64-VERIFICATION.md describes PSS-20 as using no sleep 8 and G1 OIDs NOT set --
the original empty-holders approach. The delivered `68-pss-20-all-g1-not-ready.sh` uses the
stale-path approach: primes all 12 OIDs (with sleep 8), then sets G1 OIDs to NoSuchInstance
via sim_set_oid_stale. Phase 64-03-SUMMARY correctly documents the redesign decision but
VERIFICATION.md was not updated.

**Impact:** Documentation only. Scenario correctly blocks the gate. No functional impact.

### Issue 3 (DESIGN BOUNDARY): Stage 3 scenarios not self-contained under run-all.sh

**Files:** tests/e2e/scenarios/62-pss-14 through 68-pss-20

**Description:** None of the 7 Stage 3 scenarios apply the four-tenant fixture. Under run-all.sh
without pre-applied cfg08-pss-four-tenant.yaml, all Stage 3 assertions will record_fail.

Evidence this is intentional: Phase 64-01-SUMMARY states run-all.sh is the unconditional full
suite runner with Stage 3 fixture managed by run-stage3.sh. The cross-stage summary in
run-all.sh uses a _pss_s3_ran guard, producing no output when Stage 3 has no results.

**Recommendation:** Add comment to run-all.sh header: Stage 3 scenario validation (scenarios
62-68) requires run-stage3.sh. Scenarios 62-68 will record_fail under run-all.sh unless the
four-tenant fixture is pre-applied externally.

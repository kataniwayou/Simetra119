## Detailed Findings

### 4-Watcher Consistency Pattern

All four watchers conform to the watcher-validates-registry-stores architecture from Phase 35.

| Watcher | ConfigMap Name | ConfigKey | Validates | Store | Store Method |
|---|---|---|---|---|---|
| OidMapWatcherService | simetra-oid-metric-map | oid_metric_map.json | ValidateAndParseOidMap (3-pass: dup OID, dup name, build clean dict) | OidMapService | UpdateMap(dict) |
| CommandMapWatcherService | simetra-oid-command-map | oid_command_map.json | ValidateAndParseCommandMap (3-pass: dup OID, dup name, build clean dict) | CommandMapService | UpdateMap(dict) |
| DeviceWatcherService | simetra-devices | devices.json | ValidateAndBuildDevicesAsync (CS extraction, DNS, OID resolution, dup IP+Port) | DeviceRegistry | ReloadAsync(List<DeviceInfo>) |
| TenantVectorWatcherService | simetra-tenants | tenants.json | ValidateAndBuildTenants (structural, Role, MetricName, IP+Port, TEN-13 gate) | TenantVectorRegistry | Reload(TenantVectorOptions) |

All four watchers are registered in DI as AddSingleton<T> + AddHostedService(sp => sp.GetRequiredService<T>()) inside the IsInCluster() branch of ServiceCollectionExtensions.cs. The pattern is consistent across all four. Both registries (DeviceRegistry, TenantVectorRegistry) contain no validation logic — confirmed pure stores with atomic FrozenDictionary swap.

### Cross-Phase Integration Points

#### 1. DeviceRegistry pure store / DeviceWatcherService validation (Phase 34 -> Phase 35)

Status: CONNECTED.

DeviceWatcherService.ValidateAndBuildDevicesAsync (internal static, line 239 of DeviceWatcherService.cs) performs all Phase 34 validation:
- CommunityString extraction via CommunityStringHelper.TryExtractDeviceName
- Async DNS resolution via Dns.GetHostAddressesAsync
- OID resolution via BuildPollGroups, which calls oidMapService.ResolveToOid per MetricName
- Duplicate IP+Port check (hard error, entry skipped)
- Duplicate CommunityString check across different IP+Port pairs (warning, both entries loaded)

The validated result is passed to _deviceRegistry.ReloadAsync(deviceInfos). DeviceRegistry.ReloadAsync (line 51 of DeviceRegistry.cs) accepts List<DeviceInfo>, builds two FrozenDictionary lookups (_byIpPort keyed by configAddress:port, _byName keyed by deviceName), atomically swaps both, and returns added/removed key sets. No validation inside the registry.

The validation-in-watcher, pure-store-in-registry separation from Phase 35 is fully realized.

#### 2. TenantVectorRegistry pure store / TenantVectorWatcherService validation (TEN-13, Role, MetricName)

Status: CONNECTED.

TenantVectorWatcherService.ValidateAndBuildTenants (internal static, line 83 of TenantVectorWatcherService.cs) validates per-metric entry:
1. Empty Ip check
2. Port range check [1, 65535]
3. Empty MetricName check
4. Role validation: must be Evaluate or Resolved
5. MetricName existence via oidMapService.ContainsMetricName (TEN-05)
6. IP+Port existence via deviceRegistry.TryGetByIpPort (TEN-07)

Per-command entry validation:
1. Empty Ip, Port range, empty CommandName
2. ValueType: must be Integer32, IpAddress, or OctetString (TEN-03)
3. Empty Value check
4. IP+Port existence via deviceRegistry.TryGetByIpPort (TEN-07)

TEN-13 completeness gate (lines 250-261 of TenantVectorWatcherService.cs): after per-entry validation, the entire tenant is skipped if resolvedCount == 0, evaluateCount == 0, or cleanCommands.Count == 0. All unmet conditions are reported in one structured log message.

HandleConfigMapChangedAsync also runs _validator.Validate(null, options) (structural TenantVectorOptionsValidator, line 405) before calling ValidateAndBuildTenants. Two-layer validation: structural first, then semantic.

TenantVectorRegistry.Reload (line 55 of TenantVectorRegistry.cs) accepts pre-validated TenantVectorOptions and builds priority groups and routing index. No validation. Carries over existing slot values via RoutingKey lookup.

### E2E Flow Traces

#### Flow 1: Device config hot-reload

    ConfigMap change (simetra-devices / devices.json)
    -> DeviceWatcherService watch loop (Added/Modified event)
    -> HandleConfigMapChangedAsync: deserialize to List<DeviceOptions>
    -> _reloadLock.WaitAsync
    -> ValidateAndBuildDevicesAsync: CS extraction, DNS, OID resolution, dup checks
    -> _deviceRegistry.ReloadAsync(deviceInfos): FrozenDictionary atomic swap
    -> _pollScheduler.ReconcileAsync(AllDevices): Quartz job reconciliation

Status: COMPLETE. All steps present and wired.

#### Flow 2: Tenant config hot-reload

    ConfigMap change (simetra-tenants / tenants.json)
    -> TenantVectorWatcherService watch loop (Added/Modified event)
    -> HandleConfigMapChangedAsync: deserialize to TenantVectorOptions
    -> _validator.Validate: structural validation gate
    -> _reloadLock.WaitAsync
    -> ValidateAndBuildTenants: Role/MetricName/IP+Port per-entry + TEN-13 completeness gate
    -> _registry.Reload(cleanOptions): FrozenDictionary routing index swap + slot carry-over

Status: COMPLETE. TEN-13 gate at lines 250-261 of TenantVectorWatcherService.cs. Slot carry-over in TenantVectorRegistry.Reload lines 57-120.

#### Flow 3: Poll flow (MetricPollJob -> SNMP GET -> pipeline -> tenant routing)

    Quartz trigger -> MetricPollJob.Execute
    -> _deviceRegistry.TryGetByIpPort(configAddress, port, out device)
    -> device.PollGroups[pollIndex].Oids -> Variable list
    -> IPEndPoint(IPAddress.Parse(device.ResolvedIp), device.Port)
    -> new OctetString(device.CommunityString)  [Phase 33 field consumed here]
    -> _snmpClient.GetAsync(VersionCode.V2, endpoint, community, variables, ct)
    -> DispatchResponseAsync: SnmpOidReceived { AgentIp = device.ResolvedIp }
    -> _sender.Send -> MediatR: Logging->Exception->Validation->OidResolution->ValueExtraction->TenantVectorFanOut
    -> TenantVectorFanOutBehavior: _registry.TryRoute(ip, port, metricName)

Status: COMPLETE. device.CommunityString (Phase 33) consumed at MetricPollJob line 86. device.ResolvedIp used for SNMP call and AgentIp, matching resolved IPs in TenantVectorRegistry.

#### Flow 4: Trap flow (SnmpTrapListenerService -> pipeline -> tenant routing)

    UDP datagram -> SnmpTrapListenerService.ProcessDatagram
    -> CommunityStringHelper.TryExtractDeviceName(receivedCommunity, out deviceName)
       [invalid: Debug log + TrapAuthFailed increment, continue]
    -> VarbindEnvelope -> _trapChannel.Writer.TryWrite
    -> ChannelConsumerService -> SnmpOidReceived -> MediatR pipeline
    -> TenantVectorFanOutBehavior routes by (ip, port, metricName)

Status: COMPLETE. No IDeviceRegistry dependency. Architectural constraint enforced.

#### Flow 5: Invalid config (bad CommunityString -> skip, system continues)

    DeviceOptions with CommunityString missing Simetra. prefix

Status: COMPLETE. Per-entry skip semantics confirmed. Error-level logging correct for dropped monitored device.

Severity: Low. No functional break. Recommend adding simetra-oid-command-map to the projected volumes for consistency.

### Minor Observations (Non-Breaking)

**OBS-01: Local dev tenant loading skips structural validator.**
Program.cs lines 114-133 call ValidateAndBuildTenants without first calling _validator.Validate. K8s path runs TenantVectorOptionsValidator before the semantic validator. Not a crash risk. Low priority fix: add validator call before ValidateAndBuildTenants in Program.cs for parity.

**OBS-02: DeviceWatcherService class summary understates OidMapService dependency.**
Summary says it does not touch OidMapService. Accurate for writes. The service reads from it in BuildPollGroups. Suggested clarification: reads for OID resolution but does not reload or modify it. Documentation only.

**OBS-03: simetra-oid-command-map not in deployment.yaml projected volumes.**
See MISSING-01. Operational inconsistency. No functional impact.

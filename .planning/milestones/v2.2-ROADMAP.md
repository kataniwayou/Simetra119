## Milestone Summary

**Key Decisions:**
- run-stage2.sh is PSS-only (53-61), not full suite — avoids non-PSS failures contaminating PSS stage gate
- Stage gate checks raw FAIL_COUNT (not delta) — runner only sources PSS scenarios before gate
- Explicit scenario list in runner (not glob) — prevents unexpected file pickup
- run-stage3.sh manages Stage 3 fixture lifecycle (not individual scenarios) — 7 scenarios share identical setup
- G2 assertions use tier=3 (not just tier=) — proves gate passed AND G2 reached Healthy
- Re-prime at scenario start (not OID reset at end) — ensures clean state regardless of prior scenario
- --since must exactly match observation sleep — 12s>10s created 2s overlap allowing prior-cycle log bleed
- Runner overrides _REPORT_CATEGORIES locally — generate_report reads global at call time

**Issues Resolved:**
- Stale filenames in run-stage2.sh (3 scenarios renamed during Phase 62 but Stage 2 runner not updated)
- PSS-18c/19c/17c/20c flaky log-absence assertions (--since=12s with sleep 10 created 2s overlap)
- Standalone runners skipping PSS report category (indices 52-67 exceed array when starting at 0)
- Missing cleanup trap in run-stage2.sh (no OID override reset on unexpected exit)

**Issues Deferred:**
- PSS-03 (stale synthetic source): Requires synthetic-sourced OID mapping in tenant fixture
- PSS-04/05 (trap/command immunity): Requires trap/command-sourced OID mapping in tenant fixture

**Technical Debt Incurred:**

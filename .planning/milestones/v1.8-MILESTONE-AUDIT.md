# v1.8 Combined Metrics — Milestone Audit

**Milestone Goal:** Allow operators to configure aggregate (combined) metrics that compute a
single synthetic value (sum/subtract/absDiff/mean) from multiple raw SNMP OIDs per poll cycle,
dispatching the result through the full MediatR pipeline so it appears in Prometheus as a normal
snmp_gauge and is countable via snmp_aggregated_computed_total.

## E2E Flow Traces

### Flow 1: Valid Config -> Prometheus snmp_gauge (Happy Path)

Step 1: Operator sets AggregatedMetricName and Aggregator in devices.json ConfigMap.
    PollOptions.AggregatedMetricName = totalTraffic
    PollOptions.Aggregator = sum

Step 2: DeviceWatcherService.HandleConfigMapChangedAsync fires on ConfigMap watch event.
    Calls ValidateAndBuildDevicesAsync -> BuildPollGroups.

Step 3: BuildPollGroups validates 5 rules (all pass for this config).
    co-presence: both fields set
    Enum.TryParse(sum) -> AggregationKind.Sum
    resolvedOids.Count >= 2
    name not in seenAggregatedNames
    !oidMapService.ContainsMetricName(totalTraffic)
    Builds: AggregatedMetricDefinition(totalTraffic, Sum, [oid1, oid2])

Step 4: MetricPollInfo.AggregatedMetrics = [AggregatedMetricDefinition(...)].
    DeviceRegistry stores DeviceInfo with populated poll group.

Step 5: MetricPollJob.Execute fires on Quartz schedule.
    Calls SnmpClient.GetAsync for [oid1, oid2].
    Response: [Variable(oid1, Gauge32(1000)), Variable(oid2, Gauge32(500))]

Step 6: DispatchResponseAsync dispatches oid1 and oid2 as individual Poll messages (existing flow),
    then iterates pollGroup.AggregatedMetrics.

Step 7: DispatchAggregatedMetricAsync:
    Builds oidValues dict: {oid1->1000, oid2->500}
    Validates both OIDs present and numeric.
    Computes: Sum([1000, 500]) = 1500
    TypeCode: Gauge32 (Sum -> Gauge32 per SelectTypeCode)
    Value: new Gauge32((uint)Clamp(1500, uint.MinValue, uint.MaxValue))

Step 8: Synthetic SnmpOidReceived constructed and dispatched:
    Oid = 0.0  (sentinel, passes ValidationBehavior regex)
    Source = SnmpSource.Synthetic
    MetricName = totalTraffic  (pre-set at construction)
    DeviceName = device.Name  (required, passes ValidationBehavior DeviceName check)
    TypeCode = Gauge32
    PollDurationMs = null  (no round-trip for synthetic)

Step 9: MediatR pipeline (Logging -> Exception -> Validation -> OidResolution -> ValueExtraction -> TenantVectorFanOut -> OtelMetricHandler):
    ValidationBehavior:          0.0 passes regex; DeviceName non-null — PASS
    OidResolutionBehavior:       Source==Synthetic -> bypass; MetricName=totalTraffic preserved — PASS
    ValueExtractionBehavior:     TypeCode=Gauge32 -> ExtractedValue=1500 — PASS
    TenantVectorFanOutBehavior:  routes by (ip, device.Port, totalTraffic) if tenant configured — PASS
    OtelMetricHandler:           RecordGauge(totalTraffic, 0.0, deviceName, ip, synthetic, gauge32, 1500) — PASS

Step 10: Prometheus scrape exposes:
    snmp_gauge{metric_name=totalTraffic, oid=0.0, device_name=router-1, source=synthetic, snmp_type=gauge32} 1500

Step 11: PipelineMetricService.IncrementAggregatedComputed(device.Name)
    snmp_aggregated_computed_total{device_name=router-1} increments by 1.

STATUS: COMPLETE — no breaks at any step.

### Flow 2: Invalid Config -> Error Log, Individual OIDs Still Poll

All four sub-cases share the same guarantee: the validation block in BuildPollGroups sets
aggregatedMetric = null (never assigns an AggregatedMetricDefinition) and does NOT call
continue — result.Add always executes, so the poll group loads for individual OID polling.

Sub-case A: Partial config — AggregatedMetricName set but Aggregator empty.
    hasName=true, hasAggregator=false -> enters !hasName || !hasAggregator branch.
    LogError: combined metric has Aggregator missing.
    aggregatedMetric = null -> AggregatedMetrics = [].
    Poll group loaded. AggregatedMetrics empty -> foreach in DispatchResponseAsync never executes.
    STATUS: COMPLETE

Sub-case B: Invalid Aggregator string (e.g., average, not a valid AggregationKind).
    Enum.TryParse<AggregationKind>(average, ignoreCase:true) -> false.
    LogError: invalid Aggregator 'average'.
    aggregatedMetric = null. Individual polling unaffected.
    STATUS: COMPLETE

Sub-case C: AggregatedMetricName collides with OID map entry.
    oidMapService.ContainsMetricName(sysUpTime) -> true.
    LogError: name collides with existing OID map entry.
    aggregatedMetric = null. Individual polling unaffected.
    STATUS: COMPLETE

Sub-case D: OID absent from response at poll time (runtime, not config-load).
    DispatchAggregatedMetricAsync: oidValues.TryGetValue(missingOid) -> false.
    LogWarning: Combined metric skipped — OID absent from response.
    Early return; snmp.aggregated.computed NOT incremented for this cycle.
    Individual varbinds already dispatched before the aggregate block runs — unaffected.
    STATUS: COMPLETE

### Flow 3: Tenant Routing of Synthetic Metric

Step 1: Synthetic SnmpOidReceived enters TenantVectorFanOutBehavior.
Step 2: metricName = msg.MetricName = totalTraffic (pre-set; preserved by OidResolution bypass).
Step 3: metricName not null and not OidMapService.Unknown -> routing proceeds.
Step 4: msg.DeviceName != HeartbeatDeviceName -> deviceRegistry.TryGetDeviceByName branch.
Step 5: TryGetDeviceByName(deviceName, out device) succeeds -> device.Port obtained.
Step 6: _registry.TryRoute(ip, device.Port, totalTraffic, out var holders).
    Routing key: (ip, port, totalTraffic).
    If a tenant has a slot configured for totalTraffic -> holder.WriteValue(1500, ...) executes.
    If no tenant configures totalTraffic -> TryRoute returns false; silently skipped.
Step 7: next() is called OUTSIDE the try/catch block -> OtelMetricHandler fires unconditionally.

STATUS: COMPLETE — synthetic MetricName routes correctly via existing (ip, port, metricName) key.
NOTE: If no tenant vector is configured for the aggregate metric name, the metric still appears
in Prometheus via OtelMetricHandler. Tenant routing is optional and best-effort by design.

## Summary

The v1.8 Combined Metrics milestone is integration-complete. All four phases connect without
breaks:

- Phase 37 types are consumed exactly where expected in Phases 38 and 40.
- Phase 38 validation runs at config load time, never at poll time — validation failure is
  non-destructive to individual OID polling.
- Phase 39 bypass guard correctly preserves the pre-set MetricName through OidResolutionBehavior
  for all synthetic messages. Sentinel OID 0.0 passes ValidationBehavior without any change
  to that behavior.
- Phase 40 dispatches synthetic messages through the full MediatR pipeline with correct TypeCode
  selection (Subtract/AbsDiff -> Integer32, Sum/Mean -> Gauge32), increments the counter only on
  successful dispatch, and isolates aggregate exceptions so individual poll success is unaffected.
- The snmp.aggregated.computed counter flows correctly to the Grafana operations dashboard via
  the standard OTel -> Prometheus naming transformation (dots to underscores, _total suffix).

The only finding is documentation drift in Phase 37 and 38 SUMMARY files: the runtime record is
named CombinedMetricDefinition in the documentation but AggregatedMetricDefinition in all source
and test files. This has no runtime impact.

## Milestone Summary

**Key Accomplishments:**
1. 6 percentage gauges (tenant.metric.stale.percent, etc.) replacing 6 counters — gauges record 0-100% per cycle
2. Gather-then-decide EvaluateTenant flow — gather all tier results, decide state, dispatch only when Unresolved, record all metrics at single exit
3. Resolved metric direction flipped — now measures violated holders (higher = worse), consistent with evaluate
4. State renamed: tenant.state → tenant.evaluation.state (consistent namespace)
5. Dashboard columns show percentage values with (%) suffix, direct gauge PromQL queries
6. 7 E2E scenarios (107-113) including partial percentage (50%) verification

**Key Decisions:**
- Gauge naming: tenant.metric.* for holders, tenant.command.* for commands, tenant.evaluation.* for state/duration
- Dispatch after decide (quick-088): commands only fire when state = Unresolved, not when evaluate gate fires pre-decision
- Stale path: records stale% + command% but NOT resolved%/evaluate% (stale data unreliable)
- NotReady: only state + duration (no percentage gauges)
- Zero denominator: record 0.0 (validation prevents in practice)
- CommandWorkerService no longer holds ITenantMetricService — all tenant metrics in SnapshotJob

**Issues Resolved:**
- Suppressed%=100 on Resolved tenants (dispatch ran before state decision — fixed by quick-088)
- Gauge export investigation: Gauge<double> works correctly, suppressed was genuinely 100% from dispatch-before-decide ordering

**Technical Debt:**
- Scenario 113 has misleading comment and unreachable poll_until_log target (wastes 30s, assertions correct)
- Gauge-based E2E assertions inherently flaky due to Prometheus scrape timing
